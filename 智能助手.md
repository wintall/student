# 智能助手二阶段落地建议

## 1. 产品定位

二阶段建议将系统升级为“校园 AI 助手中心”。

前端以一个足球形态的悬浮气泡作为统一入口。用户点击足球后展开问答面板，像豆包类产品一样通过自然语言完成提问、查询、分析和辅助决策。

核心原则：

- 不把能力按钮做成传统页面跳转。
- 能力按钮只用于切换助手能力、快捷问题、工具范围和提示词上下文。
- 用户也可以不点按钮，直接自然语言提问，由系统自动识别意图并调用对应能力。
- 管理员、教师、学生都使用同一个助手入口，但后端根据角色和权限返回不同能力与数据范围。

## 2. 能力模块

建议第一版能力组件如下：

| 模块 | 定位 | 典型问题 |
| --- | --- | --- |
| 自动 | 默认模式，自动判断用户意图 | “我最近数学退步了怎么办？” |
| RAG 知识问答 | 文学、制度、知识库检索问答 | “三打白骨精是哪一回？” |
| 教务工具调用 | 成绩、课表、公告、请假等系统数据查询 | “我今天有什么课？” |
| 学习辅导 | 学习计划、复习建议、课程辅导 | “帮我制定英语复习计划” |
| 文档处理 | 上传文档后的总结、提纲、重点提取 | “帮我总结这份课程资料” |
| 情绪陪伴 | 考试压力、人际烦恼、学习挫败疏导 | “我最近考试压力很大” |
| 数据分析 | 数据体检、统计、趋势分析、管理建议 | “系统还有哪些高危异常？” |
| 世界杯问答 | 世界杯知识、球队、球星、规则问答 | “2022 世界杯冠军是谁？” |
| AI 知识问答 | LangChain、LangGraph、FastAPI、Dify、Python、Linux、SQL 等技术知识 | “LangGraph 和 LangChain 有什么区别？” |
| 更多 | 后续扩展入口 | 新能力插件、外部工具、MCP 服务 |

## 3. 推荐技术架构

建议采用“统一入口 + Agent 调度 + 工具注册表 + 可扩展知识库”的架构。

```text
足球气泡入口
  -> 校园助手聊天面板
  -> /api/v1/campus-agent/chat
  -> CampusAgent
      -> 身份识别
      -> 意图识别
      -> 权限判断
      -> 工具选择
      -> LangGraph 编排
      -> RAG / Deep Agents / MCP / 系统工具
      -> 回复生成
      -> 会话与工具调用记录
```

### 3.1 LangChain

用途：

- LLM 接入封装。
- Prompt 模板管理。
- 工具定义。
- 结构化输出。
- 普通问答、学习辅导、情绪陪伴的基础能力。

建议：

- 不让 LangChain 直接裸操作数据库。
- 数据库访问必须通过后端定义好的安全工具。

### 3.2 LangGraph

用途：

- 作为核心 Agent 编排框架。
- 管理多轮状态、意图路由、工具调用、权限分支和回复生成。

建议：

- 二阶段主 Agent 以 LangGraph 为核心。
- 每次请求进入图流程，由节点完成：
  - 用户身份与角色上下文加载。
  - 意图识别。
  - 安全工具选择。
  - 工具执行。
  - RAG 检索。
  - 回复合成。

### 3.3 Deep Agents

用途：

- 复杂多步骤任务。
- 文档处理。
- 报告生成。
- AI 知识总结。
- 多文件、多轮、多阶段分析。

建议：

- 不要所有请求都走 Deep Agents。
- 只在文档处理、深度研究、复杂数据分析这类任务中启用。

### 3.4 MCP

用途：

- 作为后续工具扩展协议。
- 连接外部工具、知识服务、文件系统、搜索工具或代码工具。

建议：

- 初期不要把所有内部工具都 MCP 化。
- 先做 1 到 2 个示范 MCP 工具即可，例如：
  - 天气工具。
  - AI 知识库检索工具。
  - 文档摘要工具。

## 4. 后端接口设计方向

建议统一主接口：

```http
POST /api/v1/campus-agent/chat
```

请求示例：

```json
{
  "session_id": "xxx",
  "mode": "auto",
  "message": "我最近考试压力很大，数学成绩也下降了"
}
```

响应示例：

```json
{
  "reply": "我能感觉到你最近压力不小...",
  "mode": "emotional_support",
  "intent": "emotion_and_study",
  "tool_calls": [
    {
      "tool": "get_my_scores",
      "status": "success"
    }
  ],
  "references": [],
  "suggested_questions": [
    "帮我制定数学复习计划",
    "查看我的最近成绩趋势"
  ]
}
```

## 5. 工具注册表设计

所有真实业务操作必须通过工具注册表执行，不允许大模型直接操作数据库。

工具字段建议：

| 字段 | 说明 |
| --- | --- |
| tool_name | 工具名称 |
| description | 工具说明 |
| allowed_roles | 允许角色 |
| required_permission | 所需权限 |
| scope_rule | 数据范围规则 |
| risk_level | 风险等级 |
| requires_confirmation | 是否需要二次确认 |
| handler | 后端执行函数 |

示例工具：

| 工具 | 能力 | 权限原则 |
| --- | --- | --- |
| get_my_scores | 查询本人成绩 | 学生只能查自己 |
| get_student_scores | 查询学生成绩 | 教师按授课或班主任范围，管理员全局 |
| get_my_schedule | 查询本人课表 | 学生/教师查自己 |
| get_pending_leave_reviews | 查询待审批请假 | 班主任、院系主任、管理员 |
| get_recent_announcements | 查询公告 | 登录用户可用 |
| data_health_summary | 数据体检摘要 | 管理员或授权角色 |
| search_literature_knowledge | 文学 RAG 检索 | 登录用户可用 |
| search_ai_knowledge | AI 知识问答 | 登录用户可用 |
| search_worldcup_knowledge | 世界杯问答 | 登录用户可用 |
| summarize_document | 文档总结 | 登录用户可用，按上传文件归属隔离 |

## 6. 权限与安全原则

- 学生只能查询自己的成绩、课表、请假、通知。
- 教师只能查询自己授课、班主任或被授权范围内的数据。
- 管理员可以查询全局统计、数据体检和系统数据。
- 情绪陪伴不做医疗诊断。
- 严重心理风险提示用户联系老师、家长、心理老师或专业机构。
- 文档处理只处理用户自己上传或有权限访问的文件。
- 高风险操作需要二次确认。
- 所有工具调用要记录日志，便于审计和答辩演示。

## 8. 开发顺序建议

### 第一阶段：方案与文档

- 明确校园助手产品定位。
- 完成二阶段需求文档。
- 完成表设计文档。
- 完成接口设计文档。
- 完成开发计划文档。
- 建立开发日志。

### 第二阶段：前端统一入口

- 将现有 AI 气泡改为足球入口。
- 展开后显示聊天面板。
- 输入框下方加入能力组件。
- 点击组件只切换模式、快捷问题和提示词上下文，不跳转页面。

### 第三阶段：统一后端 Agent 接口

- 新增统一聊天接口。
- 设计会话和消息记录。
- 实现 mode 与 intent 识别。
- 接入角色和权限上下文。

### 第四阶段：系统内工具调用

- 接入成绩查询。
- 接入课表查询。
- 接入请假查询和审批提醒。
- 接入公告通知。
- 接入数据体检摘要。

### 第五阶段：知识库与 RAG

- 复用现有文学 RAG。
- 新增 AI 知识库。
- 新增世界杯知识库。
- 统一返回引用来源。

### 第六阶段：复杂能力

- 文档处理接入 Deep Agents。
- 复杂数据分析接入 LangGraph 多步骤流程。
- MCP 做 1 到 2 个示范工具。

### 第七阶段：验收与演示

- 准备管理员、教师、学生三个演示账号。
- 准备每种模式的示例问题。
- 完成测试记录。
- 完成二阶段总结文档。

## 9. 文档编写建议

建议采用“开发前先写框架，开发中持续更新，开发后补充最终版”的方式。

不要等全部开发完再写文档。二阶段强调 AI 产品思维，如果开发前没有需求和设计文档，后面容易做散。

推荐顺序：

1. 先写需求文档和开发计划文档，确定范围。
2. 再写表设计文档和接口设计文档，指导开发。
3. 开发过程中维护开发日志，记录变更和问题。
4. 开发完成后回填最终字段、接口示例、截图说明和验收结果。

这样文档既能指导开发，也能成为最后交付材料。
