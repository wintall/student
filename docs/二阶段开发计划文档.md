# 二阶段开发计划文档：校园 AI 助手中心

版本：v1.0  
日期：2026-06-30  
状态：二阶段交付版

## 1. 总体策略

二阶段采用“先文档、后骨架、再能力、最后打磨”的策略。

不建议一次性把 LangChain、LangGraph、Deep Agents、MCP、所有知识库、所有工具全部做满。更合理的方式是：

1. 先完成统一助手入口和接口骨架。
2. 先接入最稳定的系统内工具。
3. 再逐步增强 RAG、文档处理、MCP 等能力。

## 4. 考勤模块开发建议

建议新增 `attendance_record` 表。

原因：

- 请假是申请审批，考勤是出勤事实，不能混成一张表。
- 考勤后续可能来自手工录入、请假同步、导入、定位签到、课程签到。
- 独立表更利于统计、查询和权限控制。

第一版建议做轻量考勤：

- 不做复杂打卡规则。
- 不做定位签到。
- 先支持人工维护和请假同步。
- 支持日考勤或课程考勤预留。

## 5. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| AI 功能过多导致开发失控 | 先做 P0，再扩 P1/P2 |
| 大模型误查越权数据 | 所有工具强制权限校验 |
| 文档处理耗时长 | 异步任务或加载态 |
| RAG 依赖向量库不稳定 | 保留普通问答/问答对兜底 |
| 前端助手遮挡页面 | 支持拖拽、收起、位置记忆 |
| 考勤规则过复杂 | 第一版只做记录和审批联动 |

## 7. 足球助手自然语言 CRUD 开发流程规范

### 7.1 总体原则

足球助手的系统内操作必须遵循“白名单工具 + 后端硬权限 + 数据范围控制 + 高风险确认 + 调用审计”的流程。

禁止事项：

- 禁止让大模型直接生成 SQL 并执行。
- 禁止只凭前端菜单控制权限。
- 禁止只用 `campus-agent:access` 作为所有工具的权限。
- 禁止未确认就执行删除、批量修改、敏感字段修改等高风险操作。
- 禁止返回当前用户权限范围外的数据。

### 7.2 工具接入流程

每新增一个自然语言工具，必须按以下顺序开发：

1. 在工具注册表中登记工具：
   - 工具编码，例如 `query_student`。
   - 工具名称，例如“查询学生”。
   - 所属模块，例如 `people`。
   - 动作类型，例如 `query/create/update/delete`。
   - 绑定权限码，例如 `people:student:list`。
   - 风险等级，例如 `low/medium/high`。
   - 是否需要二次确认。
2. 编写参数 schema：
   - 明确必填字段。
   - 明确可选字段。
   - 明确枚举值和日期格式。
3. 编写工具 handler：
   - 只能调用现有 service 层。
   - 不直接拼接 SQL。
   - 查询必须限制 `page_size`。
   - 修改和删除必须先读取目标数据用于确认和审计。
4. 编写权限和数据范围校验：
   - 先校验权限码。
   - 再校验角色可见范围。
   - 学生默认只能访问自己的数据。
   - 教师只能访问自己任课、班主任、院系权限范围内的数据。
   - admin 允许全局操作。
5. 接入 `CampusAgentExecutor`：
   - 统一执行权限校验。
   - 统一处理缺参提示。
   - 统一处理二次确认。
   - 统一格式化返回。
6. 写入开发日志：
   - 新增工具列表。
   - 权限码。
   - 风险等级。
   - 测试账号和测试结果。

### 7.3 风险等级

| 风险等级 | 说明 | 示例 | 处理方式 |
| --- | --- | --- | --- |
| low | 只读查询，不改变数据 | 查询我的课表、查询公告 | 权限通过后直接执行 |
| medium | 创建记录或修改非敏感状态 | 提交请假、发布公告、修改教室容量 | 建议确认，重要业务必须确认 |
| high | 删除、批量操作、敏感字段修改 | 删除学生、删除课程、批量改成绩 | 必须二次确认 |

### 7.4 第一阶段工具范围

第一阶段只完成工具白名单、权限映射和低风险查询骨架，不开放直接删除。

优先工具：

| 工具编码 | 功能 | 权限码 | 风险 |
| --- | --- | --- | --- |
| query_my_schedule | 查询我的课表 | `schedule:my:list` | low |
| query_my_attendance | 查询我的考勤 | `attendance:my:list` | low |
| query_my_leave | 查询我的请假 | `leave:request:list` | low |
| create_leave_request | 提交请假申请 | `leave:request:create` | medium |
| query_announcements | 查询公告 | `announcement:list:view` | low |
| query_student | 查询学生 | `people:student:list` | low |
| query_teacher | 查询教师 | `people:teacher:list` | low |
| query_course | 查询课程 | `teaching:course:list` | low |
| query_class | 查询班级 | `org:clazz:list` | low |

### 7.5 二次确认规范

当工具 `confirm_required=true` 时，第一次请求只返回确认信息，不执行真实写操作。

返回内容必须包含：

- 操作类型。
- 目标对象。
- 关键字段变更。
- 风险提示。
- 确认口令或确认按钮所需的 `pending_action_id`。

用户确认后，后端根据 `pending_action_id` 再执行工具，不能依赖前端重新传完整参数。

### 7.6 验证规范

每个工具至少用三类账号验证：

- admin：验证完整权限。
- teacher / counselor / department_head：验证管理范围。
- student：验证只能看自己的数据。

每次验证至少覆盖：

- 有权限成功。
- 无权限拒绝。
- 参数缺失提示。
- 查询结果不越权。
- 高风险操作不会直接执行。

## 9. 教务操作能力开发规划

教务操作不是传统菜单页面，而是足球助手内部的一个能力模式。用户切换到该能力后，可以用自然语言发起权限范围内的系统操作。后端仍然必须经过工具白名单、RBAC 权限校验、数据范围控制、确认流程和操作日志。

### 9.1 第一阶段：安全查询

目标是让 admin 和具备相应权限的管理角色，先通过自然语言查询核心教务对象，不改变任何业务数据。

本阶段接入工具：

| 工具编码 | 功能 | 权限码 | 风险 |
| --- | --- | --- | --- |
| query_student | 查询学生 | `people:student:list` | low |
| query_teacher | 查询教师 | `people:teacher:list` | low |
| query_course | 查询课程 | `teaching:course:list` | low |
| query_class | 查询班级 | `org:clazz:list` | low |

开发要求：

- 只调用现有 service 层，不直接拼接 SQL。
- 查询结果限制条数，默认返回前 8 条。
- 关键词为空时允许返回少量列表，但必须限制数量。
- 返回内容以摘要为主，避免在聊天窗口暴露过多敏感字段。
- 每次工具调用都写入短期会话记忆，便于后续追问。

### 9.2 第二阶段：新增操作

查询稳定后再开放新增类工具，例如新增学生、教师、课程、班级、教室、课表、公告。新增操作必须先补齐必填参数，再生成确认摘要，用户确认后才执行。

确认机制要求：

- 后端创建 `agent_pending_action` 待确认动作，不依赖前端重新提交完整参数。
- 第一次识别到写操作时，只保存工具码、参数、摘要、风险等级和过期时间。
- 助手回复中展示待确认内容和 `pending_action_id`。
- 用户说“确认”或点击确认后，后端按 `pending_action_id` 读取保存参数并再次校验权限。
- 执行成功后将待确认动作标记为 `executed`。
- 执行失败、过期或取消时标记对应状态。
- 写操作结果写入 `operation_log`，便于后续审计。

第一批新增工具只做 `create_student`：

| 工具编码 | 功能 | 权限码 | 风险 | 确认 |
| --- | --- | --- | --- | --- |
| create_student | 新增学生 | `people:student:create` | medium | required |

`create_student` 参数规则：

- 必填：姓名、学号、性别、身份证号、班级。
- 可选：手机号、邮箱、入学日期。
- 班级支持按班级名称或班级代码匹配为 `clazz_id`。
- 匹配不到班级时追问，不创建。
- 匹配多个班级时返回候选项，不创建。
- 缺少必填字段时返回缺参提示，不创建 pending action。

### 9.3 第三阶段：修改与停用

修改必须展示修改前后对比。删除类操作默认设计为停用、归档或禁用账号，不优先做物理删除。高风险操作必须二次确认并记录审计日志。

## 10. 统一 Agent 底座开发流程

足球校园助手后续会继续接入 RAG、学习辅导、文档处理、数据分析、AI 知识问答等能力，因此 Agent 底座必须避免只服务教务 CRUD。所有能力模块统一复用“任务草稿、工具注册、权限校验、确认动作、操作日志、长期记忆”的基础设施。

### 10.1 统一流程

```text
用户消息
 -> 加载会话上下文
 -> 加载未完成任务草稿
 -> 判断新意图或续接草稿
 -> 结构化抽取参数
 -> 合并草稿参数
 -> 权限和数据范围校验
 -> 缺参则保存草稿并追问
 -> 多候选则保存候选并追问
 -> 高风险/写操作生成待确认动作
 -> 用户确认后执行工具
 -> 记录操作日志
 -> 写入短期记忆/长期记忆
```

### 10.2 优先建设顺序

1. `agent_task_draft`：正式任务草稿表，承接多轮补参和候选选择。
2. `AgentDraftService`：统一草稿读写接口，教务、RAG、文档、数据分析都通过该接口保存未完成任务。
3. `AgentMemoryService`：统一短期/长期记忆接口，先用数据库，后续接 Milvus。
4. LangGraph 状态机：把当前规则流程迁移为可扩展节点。
5. Milvus 长期记忆：保存跨会话偏好、摘要、学习轨迹和文档知识片段。

### 10.3 草稿表职责

`agent_task_draft` 只保存“未完成任务状态”，不保存已确认执行的动作。已确认动作仍由 `agent_pending_action` 管理。

草稿适用场景：

- 缺少必填参数，需要用户补充。
- 目标匹配多个候选，需要用户选择。
- 用户分多轮描述新增、修改、查询条件。
- 后续 LangGraph 节点需要恢复当前任务状态。

草稿字段建议：

```text
id
user_id
session_id
module_code
mode
tool_code
status
args_json
missing_fields_json
candidates_json
message
expires_at
created_at
updated_at
```

### 10.4 兼容后续模块

- RAG：保存未完成检索条件、书籍范围、引用偏好。
- 学习辅导：保存学习目标、课程、时间周期、输出格式。
- 文档处理：保存文件 ID、处理动作、摘要格式、待补充要求。
- 数据分析：保存分析对象、指标、筛选条件、图表偏好。
- AI 知识问答：保存技术方向、难度、题型、复习目标。

### 10.5 安全边界

- 草稿不是执行凭证，不能直接越权执行。
- 每次从草稿恢复任务后仍要重新走工具白名单和 RBAC 校验。
- 写操作仍必须生成 `agent_pending_action`，确认后才执行。
- 草稿过期后必须重新收集参数。

## 11. 二阶段最终执行路线

根据根目录 `二阶段项目需求.md`，本阶段不只是补传统后台功能，而是围绕“智能学生管理系统”体现 AI 产品思维。因此最终执行路线调整为：

```text
原始需求确认
 -> 四份交付文档草案
 -> 足球助手 UI
 -> 统一聊天接口
 -> 主 Agent + 子 Agent 路由
 -> 教务工具调用和权限保护
 -> 请假考勤模块
 -> 综合知识库 RAG
 -> 学习辅导/诗词鉴赏
 -> 文档处理/OCR/翻译
 -> 情绪陪伴和人在回路提醒
 -> 路线生活/搜索引擎/专题问答
 -> 编程助手/GitHub 助手
 -> 测试、日志和文档收口
```

## 13. 验收演示计划

### 13.1 管理员账号演示

1. 登录管理员。
2. 打开足球助手。
3. 询问“你能做什么”。
4. 查询所有学生。
5. 修改某个学生或教师字段，展示确认机制。
6. 查询数据体检摘要。
7. 打开综合知识库，询问“十常侍是谁”。
8. 进入搜索引擎模式，询问实时问题。

### 13.2 学生账号演示

1. 登录学生测试账号。
2. 询问“我是谁”。
3. 询问“我的老师是谁”。
4. 查询我的成绩、我的课表、我的考勤。
5. 说“我想请假”，演示多轮补参和确认。
6. 说“我最近学习压力大”，演示情绪陪伴。
7. 说“赏析一下将进酒”，演示学习辅导。

### 13.3 教师/班主任账号演示

1. 查询我的授课安排。
2. 查询班级学生或请假审批。
3. 查询本班/本院系考勤情况。
4. 验证不能访问越权数据。

## 15. 风险收口

| 风险 | 当前处理 |
| --- | --- |
| 自然语言理解不稳定 | 主 Agent/子 Agent 分流，规则兜底，任务草稿，多轮补参 |
| 大模型返回空内容 | RAG、数据分析、情绪陪伴等模块均设置兜底 |
| 越权数据访问 | 工具白名单 + RBAC + service 层数据范围 |
| 写操作误执行 | `agent_pending_action` 二次确认 |
| 长文本导入耗时 | 支持本地路径导入，前端超时问题通过后端处理链路优化 |
| 外部服务不可用 | 搜索、地图、GitHub 均有错误提示和降级说明 |
| 心理陪伴边界 | 不诊断，不治疗，高风险转人工 |
