# 二阶段开发计划文档：校园 AI 助手中心

版本：v0.1  
日期：2026-06-26  
状态：开发前计划，执行中持续更新

## 1. 总体策略

二阶段采用“先文档、后骨架、再能力、最后打磨”的策略。

不建议一次性把 LangChain、LangGraph、Deep Agents、MCP、所有知识库、所有工具全部做满。更合理的方式是：

1. 先完成统一助手入口和接口骨架。
2. 先接入最稳定的系统内工具。
3. 再逐步增强 RAG、文档处理、MCP 等能力。

## 2. 里程碑

### M2：足球助手前端骨架

目标：

- 将现有 AI 气泡改造成足球入口。
- 展开后显示统一聊天面板。
- 输入框下方显示能力组件。
- 点击组件切换模式、快捷问题和 placeholder。
- 不做传统页面跳转。

风险：

- 需要处理右下角 AI 气泡、左下角天气卡的视觉避让。
- 聊天面板不能遮挡过多后台操作区域。

### M3：校园 Agent 后端骨架

目标：

- 新增统一 `/campus-agent/chat` 接口。
- 新增会话、消息、工具调用记录表。
- 实现基础 mode 和 intent 识别。
- 返回统一结构：回复、意图、工具调用、引用、推荐问题。

技术建议：

- LangGraph 作为主编排。
- LangChain 用于模型、提示词和工具包装。
- 初期可先用规则意图识别，后续再升级 LLM 结构化识别。

### M4：系统内工具调用

目标：

- 接入成绩查询。
- 接入课表查询。
- 接入公告查询。
- 接入请假查询。
- 接入考勤查询。
- 接入数据体检摘要。

原则：

- 工具只返回当前用户权限范围内的数据。
- 大模型不能直接拼 SQL。
- 所有工具调用写入 `agent_tool_call`。

### M5：请假考勤模块

目标：

- 前端菜单“请假管理”调整为“请假考勤”。
- 原请假功能迁移到该模块下。
- 新增我的考勤。
- 新增学生考勤管理。
- 新增教职工考勤管理。
- 请假审批通过后联动考勤记录。

建议开发顺序：

1. 新建 `attendance_record` 表和模型。
2. 新建考勤 schema/service/api。
3. 实现我的考勤。
4. 实现学生考勤管理。
5. 实现教职工考勤管理。
6. 接入请假审批通过后的考勤同步。
7. 同步菜单和权限。

### M6：RAG 和知识库增强

目标：

- 复用现有文学 RAG。
- 新增 AI 知识问答知识库。
- 新增世界杯问答知识库。

- 世界杯问答可以先做小型知识库，避免依赖不稳定外部搜索。

### M7：文档处理和 Deep Agents

## 4. 考勤模块开发建议

建议新增 `attendance_record` 表。

原因：

- 请假是申请审批，考勤是出勤事实，不能混成一张表。
- 考勤后续可能来自手工录入、请假同步、导入、定位签到、课程签到。
- 独立表更利于统计、查询和权限控制。

第一版建议做轻量考勤：

- 不做复杂打卡规则。
- 不做定位签到。
- 先支持人工维护和请假同步。
- 支持日考勤或课程考勤预留。

## 5. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| AI 功能过多导致开发失控 | 先做 P0，再扩 P1/P2 |
| 大模型误查越权数据 | 所有工具强制权限校验 |
| 文档处理耗时长 | 异步任务或加载态 |
| RAG 依赖向量库不稳定 | 保留普通问答/问答对兜底 |
| 前端助手遮挡页面 | 支持拖拽、收起、位置记忆 |
| 考勤规则过复杂 | 第一版只做记录和审批联动 |

## 7. 足球助手自然语言 CRUD 开发流程规范

### 7.1 总体原则

足球助手的系统内操作必须遵循“白名单工具 + 后端硬权限 + 数据范围控制 + 高风险确认 + 调用审计”的流程。

禁止事项：

- 禁止让大模型直接生成 SQL 并执行。
- 禁止只凭前端菜单控制权限。
- 禁止只用 `campus-agent:access` 作为所有工具的权限。
- 禁止未确认就执行删除、批量修改、敏感字段修改等高风险操作。
- 禁止返回当前用户权限范围外的数据。

### 7.2 工具接入流程

每新增一个自然语言工具，必须按以下顺序开发：

1. 在工具注册表中登记工具：
   - 工具编码，例如 `query_student`。
   - 工具名称，例如“查询学生”。
   - 所属模块，例如 `people`。
   - 动作类型，例如 `query/create/update/delete`。
   - 绑定权限码，例如 `people:student:list`。
   - 风险等级，例如 `low/medium/high`。
   - 是否需要二次确认。
2. 编写参数 schema：
   - 明确必填字段。
   - 明确可选字段。
   - 明确枚举值和日期格式。
3. 编写工具 handler：
   - 只能调用现有 service 层。
   - 不直接拼接 SQL。
   - 查询必须限制 `page_size`。
   - 修改和删除必须先读取目标数据用于确认和审计。
4. 编写权限和数据范围校验：
   - 先校验权限码。
   - 再校验角色可见范围。
   - 学生默认只能访问自己的数据。
   - 教师只能访问自己任课、班主任、院系权限范围内的数据。
   - admin 允许全局操作。
5. 接入 `CampusAgentExecutor`：
   - 统一执行权限校验。
   - 统一处理缺参提示。
   - 统一处理二次确认。
   - 统一格式化返回。
6. 写入开发日志：
   - 新增工具列表。
   - 权限码。
   - 风险等级。
   - 测试账号和测试结果。

### 7.3 风险等级

| 风险等级 | 说明 | 示例 | 处理方式 |
| --- | --- | --- | --- |
| low | 只读查询，不改变数据 | 查询我的课表、查询公告 | 权限通过后直接执行 |
| medium | 创建记录或修改非敏感状态 | 提交请假、发布公告、修改教室容量 | 建议确认，重要业务必须确认 |
| high | 删除、批量操作、敏感字段修改 | 删除学生、删除课程、批量改成绩 | 必须二次确认 |

### 7.4 第一阶段工具范围

第一阶段只完成工具白名单、权限映射和低风险查询骨架，不开放直接删除。

优先工具：

| 工具编码 | 功能 | 权限码 | 风险 |
| --- | --- | --- | --- |
| query_my_schedule | 查询我的课表 | `schedule:my:list` | low |
| query_my_attendance | 查询我的考勤 | `attendance:my:list` | low |
| query_my_leave | 查询我的请假 | `leave:request:list` | low |
| create_leave_request | 提交请假申请 | `leave:request:create` | medium |
| query_announcements | 查询公告 | `announcement:list:view` | low |
| query_student | 查询学生 | `people:student:list` | low |
| query_teacher | 查询教师 | `people:teacher:list` | low |
| query_course | 查询课程 | `teaching:course:list` | low |
| query_class | 查询班级 | `org:clazz:list` | low |

### 7.5 二次确认规范

当工具 `confirm_required=true` 时，第一次请求只返回确认信息，不执行真实写操作。

返回内容必须包含：

- 操作类型。
- 目标对象。
- 关键字段变更。
- 风险提示。
- 确认口令或确认按钮所需的 `pending_action_id`。

用户确认后，后端根据 `pending_action_id` 再执行工具，不能依赖前端重新传完整参数。

### 7.6 验证规范

每个工具至少用三类账号验证：

- admin：验证完整权限。
- teacher / counselor / department_head：验证管理范围。
- student：验证只能看自己的数据。

每次验证至少覆盖：

- 有权限成功。
- 无权限拒绝。
- 参数缺失提示。
- 查询结果不越权。
- 高风险操作不会直接执行。
