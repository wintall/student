## 2026-06-26：足球助手升级为动态 3D 入口

### 本次目标

- 将足球助手入口从平面样式升级为动态、立体的足球。
- 保持原有点击展开、拖拽移动、能力切换和聊天面板逻辑不变。
- 避免 3D 依赖直接撑大主布局首屏包。

### 已完成

1. 前端新增 `three` 依赖，用于渲染 3D 足球。
2. `AIAssistant.vue` 的足球入口改为 `canvas` 渲染：
   - 使用 `SphereGeometry` 作为足球主体。
   - 使用五边形黑色贴片和环形缝线模拟足球纹理。
   - 增加环境光、主光和边缘光，让足球具备立体明暗。
   - 使用 `requestAnimationFrame` 持续旋转，面板展开时旋转更明显。
3. 采用 `await import('three')` 动态加载 Three.js：
   - 主布局包保持轻量。
   - Three.js 被拆成独立懒加载 chunk。
4. 增加 CSS 降级方案：
   - 浏览器不支持 WebGL 或 Three.js 加载失败时，仍显示带阴影和旋转效果的足球样式。
5. 组件销毁时释放 Three.js renderer、geometry、material 和动画帧，避免内存泄漏。

### 验证结果

- 前端 `npm run build` 通过。
- 构建结果中 `three.module-*.js` 独立拆包，`MainLayout` 未被 3D 依赖明显撑大。
- 当前仍存在 Vite/Rolldown 对第三方库 `/* #__PURE__ */` 注释的警告，以及项目既有大 chunk 警告；这些不是本次 3D 足球改造导致的阻断错误。

### 后续建议

1. 等校园助手后端能力稳定后，再考虑给足球加入状态反馈：
   - 空闲：慢速旋转。
   - 思考中：加快旋转或轻微跳动。
   - 报错：短暂红色描边。
2. 如果后续需要更真实的足球纹理，可以引入贴图资源，但当前阶段用几何贴片更轻量、也更容易维护。

## 2026-06-26：足球助手短期记忆与连续追问开发

### 本次目标

- 让足球助手具备第一版短期记忆。
- 支持基于上一轮工具的连续追问，例如：
  - “那明天呢”
  - “只看请假的”
  - “再看公告”
  - “上一条详细点”
- 第一版先复用已有 `conversation` 表，不新增迁移，降低对当前系统的影响。

### 设计决策

1. 第一版短期记忆复用 `conversation.messages`：
   - 存储最近若干轮用户消息和助手回复。
   - 额外记录 `intent`、`tool_code`、`tool_args`、`tool_status`。
   - 用于下一轮判断用户是否在追问上一轮结果。
2. 暂不把长期记忆写入 Milvus：
   - 本轮只做会话级短期记忆。
   - Milvus 长期记忆留到后续 `agent_memory` 阶段。
3. 不改变权限规则：
   - 即使追问命中了上一轮工具，也必须重新走 `CampusAgentExecutor` 权限校验。
4. 后续迁移方向：
   - 将 `conversation` 迁移为 `agent_session` + `agent_message`。
   - 将长期偏好、学习状态、常用任务写入 `agent_memory` 和 Milvus。

### 开发步骤

1. 新增校园助手会话记忆服务。
2. `/campus-agent/chat` 读取或创建会话。
3. 每次工具调用成功或失败后写入短期消息。
4. 当用户没有明确工具意图时，尝试从上一轮工具上下文推断。
5. 清空对话时同时清理该会话短期记忆。

### 安全边界

- 短期记忆只辅助理解，不作为权限依据。
- 权限、角色、业务数据仍实时查数据库。
- 不在短期记忆里额外保存密码、身份证等敏感字段。

### 已完成

1. 新增 `app/services/campus_agent/memory.py`：
   - 创建或读取校园助手会话。
   - 写入用户消息和助手回复。
   - 保存上一轮工具上下文。
   - 支持清空当前会话短期记忆。
2. 更新 `/api/v1/campus-agent/chat`：
   - 每轮聊天都会绑定 `session_id`。
   - 工具调用结果会写入短期记忆。
   - 普通 AI 回复也会写入短期记忆。
3. 增加连续追问推断：
   - 上一轮查课表后，支持“那明天呢 / 今天呢”。
   - 上一轮查考勤后，支持“只看请假的 / 那考勤呢”。
   - 上一轮查请假后，支持“只看待审批 / 已通过的呢”。
   - 上一轮查公告后，支持“再看公告 / 上一条详细点”。
4. 新增 `POST /api/v1/campus-agent/clear`：
   - 清空当前足球助手会话记忆。
5. 前端清空按钮改为调用校园助手清空接口。

### 验证结果

- 后端 `py_compile` 通过。
- 前端 `npm run build` 通过。
- 使用 `student01` 进行两轮模拟验证：
  - 第一轮：“我的课表” -> 命中 `query_my_schedule`。
  - 第二轮：“那明天呢” -> 复用上一轮工具上下文，继续命中 `query_my_schedule`。
  - 清空会话接口执行成功。

### 遗留问题

- 当前短期记忆仍复用 `conversation` 表，后续应迁移为专用 `agent_session/agent_message`。
- 当前追问推断仍是规则版，后续应接入 LangGraph 节点。
- 当前没有长期记忆和 Milvus 写入，后续再接 `agent_memory`。

 -> normalize_args

其中 `rag_knowledge_service.search()` 是综合知识库的新链路，和旧的 `rag_service.py` 不是同一套入口。足球助手当前主要调用新的综合知识库服务。

1. 扩展结构化问题识别：
   - “十常侍都是谁”可以识别。
   - “十常侍是谁”现在也可以识别。

2. 强化结构化答案：

   - 如果检索到了资料，但大模型总结没有返回内容，系统会从片段中提取和问题最相关的句子，整理成简短白话摘要。
   - 不再显示“当前大模型暂时没有生成总结”这类内部技术状态。
   - 仍保留简短出处，方便用户追溯来源。

## 2026-06-30 足球助手手动模块边界修复

### 问题背景

教师账号在自动模式下询问“我是谁”，系统能正确调用教务档案工具，返回教职工档案。但手动点击“教务助手”后再问“我是谁”，系统被 LLM 路由误判为“自我认同/心理困惑”，跳到了情绪陪伴模块，返回心理支持长文。

这个问题不是单句“我是谁”的问题，而是模块边界问题：用户手动选择某个能力模块时，系统应该尊重当前模块；只有非常明确的跨模块意图，才允许跳转到其他能力。

### 原因分析

自动模式优先走 `route_mode_v2()`，能把“我是谁”稳定识别为 `academic_ops`。手动教务模式下，后端走 `route_intent()` 和 `should_override_current_mode()`，原逻辑允许 LLM 在较低门槛下把教务模式覆盖成情绪模式，导致“我是谁”被解释成心理问题。

### 本次修复

1. 强化教务确定性路由：
   - 将“我是谁”“我的信息”“我的资料”“我的身份”“我的账号”“我的成绩”“我的老师”“我的课程”等个人教务表达加入教务信号。
   - `route_intent()` 先调用 `route_mode_v2()`，如果它明确返回 `academic_ops`，直接作为确定性教务路由。

2. 收紧手动教务模式的跨模块覆盖：
   - 用户手动处于 `academic_ops` / `academic_tools` 时，不再允许 LLM 猜测式跳出。
   - 只有确定性强信号，例如明确“赏析诗词”“路线规划”“搜索新闻”“分析项目”“我压力很大”等，才允许切换到对应模块。

3. 增加回归测试：
   - 教师账号手动教务模式下问“我是谁”，必须返回教职工档案。
   - “我是谁”必须被确定性路由到教务模块。
   - 原有学习辅导等明确跨模块意图仍可正常跳出。

## 2026-06-30 GitHub 助手公开仓库源码下载能力

- 只下载公开 GitHub 仓库源码，不自动运行代码。

### 本次实现

1. GitHub 助手新增 `download_repo` 任务：
   - 支持 “下载/拉取/克隆/clone/download + 代码/源码/仓库/GitHub” 等表达。

2. 新增公开仓库下载链路：
   - 读取仓库默认分支。
   - 下载 `https://api.github.com/repos/{owner}/{repo}/zipball/{branch}`。
   - 解压到 `E:\student\uploads\github_repos\owner__repo__branch`。

3. 下载完成后返回：
   - 仓库名、描述、默认分支。
   - 文件数和大小。
   - 本地保存路径。
   - 下一步代码体检提示：`分析项目 本地路径`。

### 安全边界

- 不执行下载仓库中的任何脚本。
- 不自动安装依赖。
- 不自动启动项目。
- 单个仓库下载/解压上限约 300MB、30000 文件。

用户在 RAG 知识问答中询问“刘备怎么请到诸葛亮”，系统错误召回了周瑜、鲁肃和诸葛亮斗智相关片段，并用这些无关片段生成兜底摘要。

根因是综合知识库 RAG 当前默认查询当前账号可访问的全部知识库，`kb_ids=None`；检索采用向量召回 + 关键词召回 + BM25 + 标题匹配融合，但对情节类问题缺少“核心人物/事件词必须同时命中”的约束，导致只包含“诸葛亮”的片段也可能排到前面。

### 本次修复

1. 增加核心查询词抽取：
   - 对三国情节类问题优先识别人物和事件词，如“刘备”“诸葛亮”“三顾茅庐”“草庐”“隆中”等。
   - 过滤“怎么/如何/请到/介绍/总结”等泛化问句词。

2. 增加精准关键词召回：
   - 在原向量召回、MySQL 关键词召回基础上，额外尝试核心词同时命中的 SQL 召回。
   - 对候选片段计算 `core_score`，提高同时包含核心人物/事件词的片段排序。

3. 强化兜底摘要边界：
   - 如果检索片段和问题核心词匹配不足，不再强行摘取无关原文做答案。
   - 改为提示“资料片段和问题关键词匹配不够，不能据此给出可靠结论”。

4. 增加 RAG 调试信息透传：
   - `answer()` 返回 `kb_ids` 和 `retrieval` 信息。
   - 足球助手短期上下文保存本轮检索范围和检索模式，便于后续追踪“调用了哪些知识库”。
